SAÉ 3.01 – Infrastructure IoT LoRaWAN

CCACCBSemestre 3
AC21.03 AC21.04 AC21.06 AC22.01 AC22.05 AC23.02 AC23.03

Contexte

Réalisé en binôme avec Abel Raquin, l'objectif de cette SAÉ était de concevoir et déployer une infrastructure IoT LoRaWAN complète pour le suivi GPS en temps réel de balises lors d'une course d'orientation dans la forêt autour de Roanne. Le défi principal était la transmission fiable de coordonnées GPS toutes les 10 secondes dans un environnement forestier dense, contraignant l'utilisation du Data Rate 0 ("Mode Forêt") pour maximiser la portée.

Savoirs mobilisés

Savoir

Protocole LoRaWAN (OTAA, DR0, EU868), architecture IoT (capteur → gateway → serveur → dashboard), cryptographie symétrique (AppKey AES-128), modèle OSI couches physique et réseau, Docker et VPS Linux.

Savoir-faire

Programmation Arduino (C++) pour SeeedStudio, débogage matériel (correction de bugs de pilotage Pin 38/Serial2), déploiement ChirpStack v4 via Docker Compose (Mosquitto, PostgreSQL, Redis) sur Digital Ocean, écriture d'un payload decoder JavaScript.

Savoir-être

Travail en binôme, répartition des tâches (développement embarqué / infrastructure serveur), rigueur dans la documentation technique, gestion du temps sur un projet en deux phases distinctes.

Tâches réalisées & Résultats

  • Phase 1 – POC sur TTN : code Arduino validé (OTAA, fix GPS, envoi encodé), visualisation sur Datacake avec décodeur JS (÷ 1 000 000).
  • Phase 2 – Infrastructure privée : VPS Digital Ocean, stack ChirpStack v4 sous Docker Compose, reconfiguration passerelle Dragino LPS8N.
  • Enregistrement du Device (DevEUI 8CF9572000065F6A) avec mode OTAA et validation de la séquence JoinRequest/JoinAccept dans l'onglet Events ChirpStack.
  • Test réel en extérieur : déplacement GPS visible en temps réel sur le dashboard Datacake (vidéo de démonstration produite).

Problèmes & Solutions

Problème #1 – Crash matériel de la carte

L'activation de la Pin 50 selon la documentation officielle SeeedStudio provoquait un plantage immédiat de la carte.

Solution

Après analyse du schéma électrique, nous avons piloté uniquement la Pin 38 pour l'alimentation GPS et utilisé Serial2 pour la communication, contournant le bug matériel.

Problème #2 – Migration vers le serveur privé

Les données reçues par ChirpStack étaient encodées (octets bruts) et illisibles directement dans Datacake.

Solution

Rédaction d'un payload decoder JavaScript divisant les octets par 1 000 000 pour retrouver les coordonnées float (latitude/longitude) et création d'un champ Geopoint dans Datacake.

SAÉ 3.02 – Application Android Client/Serveur

CCCSemestre 3
AC21.06 AC22.05 AC23.02 AC23.03 AC24.01Cyber AC24.03Cyber

Contexte

Projet individuel consistant à concevoir une application Android complète implémentant une architecture Client/Serveur. Les contraintes techniques imposées étaient fortes : gestion des protocoles TCP et UDP, multithreading pour gérer plusieurs clients simultanément, sérialisation Java des données et mise en place d'un système de contrôle d'accès basé sur les rôles (RBAC) avec authentification sécurisée. Développé sous Android Studio (Java).

Savoirs mobilisés

Savoir

Architecture Client/Serveur, protocoles TCP (fiable) vs UDP (rapide), sérialisation Java (Serializable), gestion du cycle de vie des Activity Android, sécurité RBAC.

Savoir-faire

Développement Android (Java) sous Android Studio, sockets TCP/UDP, ObjectOutputStream/ObjectInputStream, Thread séparés pour les opérations réseau, passage de données entre Activity par Intent.

Savoir-être

Rigueur dans les tests (émulateur + téléphone réel), documentation du code, respect des contraintes Android (interdiction opérations réseau sur le thread principal).

Tâches réalisées & Résultats

  • LoginActivity : authentification sécurisée avec gestion des rôles (Admin / Étudiant) — accès refusé avec message Toast en cas d'identifiant incorrect.
  • ServerActivity : serveur hybride TCP+UDP sur port 8080 avec threads dédiés par client, logs en temps réel.
  • RBAC : le bouton "Panneau Admin (Serveur)" est masqué (View.GONE) pour les étudiants, empêchant physiquement tout accès non autorisé.
  • Classe MessagePacket implements Serializable encapsulant type, user et content pour un transport réseau fidèle.
  • Tests validés : envoi UDP ("[UDP] Etu1 : Bonjour"), connexion multi-clients simultanés, scénario de mauvais mot de passe.

Problèmes & Solutions

Problème – NetworkOnMainThreadException

Android interdit les opérations réseau sur le thread principal, provoquant une NetworkOnMainThreadException lors des premières tentatives de connexion socket.

Solution

Toutes les opérations réseau (écoute serveur, envoi client) ont été déplacées dans des threads dédiés. Le serveur instancie un nouveau thread TCP par client accepté, garantissant la gestion parallèle sans bloquer l'interface.

SAÉ Cyber.03 – Infrastructure réseau sécurisée multi-services

CyberA CCACCB Semestre 3
AC21.01 AC21.02 AC21.03 AC21.04 AC21.06 AC22.01 AC22.02 AC22.05 AC23.01 AC24.01Cyber AC24.02Cyber AC24.03Cyber

Contexte

Projet individuel (Etu7) visant à moderniser l'infrastructure réseau vieillissante d'une entreprise fictive (Windows 2008/Debian 7) vers un environnement actuel et sécurisé (Windows Server 2019/Debian 11). Le projet couvrait la virtualisation sous vCenter, le routage dynamique OSPFv2, la gestion centralisée des identités (AD-DS), la mobilité Wi-Fi sécurisée (WPA2-Enterprise / RADIUS 802.1X), la publication d'applications RemoteApp (RDS) et la supervision (Grafana). Réseau : Domaine etu7.sae – LAN 172.32.68.0/24 – VLAN 8.

Savoirs mobilisés

Savoir

Active Directory (AD-DS, DNS, OU), protocole 802.1X / WPA2-Enterprise, OSPFv2 (area 0), VLAN et segmentation réseau, PKI et PEAP/MS-CHAPv2, virtualisation vCenter (Thin Provisioning), supervision Grafana/Prometheus.

Savoir-faire

Déploiement Windows Server 2019 sur vCenter, configuration AD-DS + DNS (nslookup), installation NPS (RADIUS), configuration Unifi (SSID WPA2-Enterprise + Visiteur isolé avec QoS 2 Mbps), ACL Cisco pour cloisonnement réseau invité, SSH v2 sur tous les équipements Cisco.

Savoir-être

Autonomie et prise d'initiative face aux incidents, démarche de diagnostic logique (logs, identification conflits de ports), justification des choix techniques par matrices de décision (Ubiquiti vs Cisco, Grafana vs Zabbix).

Tâches réalisées & Résultats

  • Déploiement VM Windows Server 2019 sur vCenter (Thin Provisioning, 60 Go), configuration AD-DS et DNS pour le domaine etu7.sae, organisation en Unités d'Organisation (OU).
  • Routage dynamique OSPFv2 sur deux routeurs Cisco physiques (R0 et R1), interconnexion via réseau 172.32.128.0/30, DHCP sécurisé confiné au VLAN 8.
  • Publication d'applications RemoteApp (LibreOffice Calc + Acrylic Wi-Fi) via portail RDWeb, accessible depuis une machine cliente du domaine.
  • Wi-Fi sécurisé : SSID Entreprise_Etu7 (WPA2-Enterprise, auth RADIUS/NPS avec compte AD ttproduction) + SSID Visiteur_Etu7 (isolé, 2 Mbps).
  • ACL Cisco pour cloisonnement strict réseau invité → LAN collaborateur, SSH v2 sur tous les équipements.

Problèmes & Solutions

Problème – Conflit de port sur le contrôleur Unifi

Lors du démarrage du contrôleur Unifi, une erreur empêchait son lancement : le port 8080 était déjà utilisé par un autre service sur la machine.

Solution

Identification du processus bloquant via le Gestionnaire des tâches (PID), arrêt du service concurrent, puis relance réussie du contrôleur Unifi. Cette approche méthodique (analyse des logs → identification du conflit → résolution ciblée) a évité une réinstallation inutile.

SAÉ Cyber.04 – Pentesting (Tests d'intrusion)

CyberB Semestre 3
AC22.05 AC24.01Cyber AC24.02Cyber AC24.05Cyber AC24.06Cyber AC25.01Cyber AC25.02Cyber

Contexte

SAÉ individuelle de tests d'intrusion éthiques réalisés sur des machines virtuelles volontairement vulnérables (VulnHub) dans un réseau isolé Host-Only (Kali Linux 192.168.195.3). Toutes les manipulations ont été effectuées dans un environnement hors-production. Les cibles sont : Kioptrix Level 1 (Linux/Samba) et SickOs 1.2 (lighttpd/PHP). Pour chaque machine, une méthodologie structurée a été suivie : reconnaissance → exploitation → élévation de privilèges → rapport.

Savoirs mobilisés

Savoir

CVE et CVSS (CVE-2003-0201, CVE-2017-0238), catégories OWASP (mauvaise configuration serveur, upload de fichier non restreint), mécanismes de débordement de tampon (buffer overflow), cronjobs Linux et permissions SUID, Egress Filtering.

Savoir-faire

Scan de ports Nmap (-sV, -sC, -p-), énumération web Gobuster, exploitation Metasploit (exploit/linux/samba/trans2open), upload de reverse shell PHP via curl -T (HTTP PUT), élévation de privilèges via chkrootkit 0.49 (script /tmp/update + SUID bash).

Savoir-être

Éthique professionnelle et respect strict du cadre légal (lab isolé, autorisations explicites), rédaction de rapports techniques clairs avec preuves de compromission, formulation de recommandations de sécurisation.

Tâches réalisées & Résultats

  • Kioptrix Level 1 : scan Nmap → Samba 2.2.1a identifié via Metasploit (smb_version) → exploit trans2open (CVE-2003-0201) → shell root direct → flag lu dans /var/spool/mail/root.
  • SickOs 1.2 : scan Nmap → lighttpd 1.4.28 → Gobuster → /test/ → méthode HTTP PUT activée → upload reverse shell PHP (port 443 pour contourner l'Egress Filtering) → shell www-data.
  • Élévation de privilèges (SickOs) : chkrootkit 0.49 (CVE-2017-0238) → cronjob daily → création de /tmp/update (chmod +s /bin/bash) → SUID bash → /bin/bash -p → root + flag.
  • Rédaction de rapports de pentest complets avec résumé d'attaque et recommandations de sécurité détaillées.

Problèmes & Solutions

Problème – Reverse shell bloqué (SickOs)

Après l'upload du shell PHP, la connexion netcat sur le port standard 4444 n'aboutissait jamais : la machine cible filtrait les connexions sortantes sur les ports non-standards (Egress Filtering/pare-feu sortant).

Solution

Modification du script PHP pour utiliser le port 443 (HTTPS), considéré comme légitime pour le trafic sortant et donc non filtré. Écoute avec sudo nc -lvnp 443 (nécessite les droits root car port < 1024) → connexion réussie.

Difficultés & Apprentissages

Difficulté – Complexité de l'interface FortiGate

La multitude d'options de l'interface FortiOS et la subtilité de l'ordre des règles de sécurité (first-match) rendait difficile la mise en place des politiques sans bloquer le trafic légitime.

Apprentissage tiré

L'importance de tester chaque règle individuellement avant de les combiner, et d'utiliser les logs FortiGate pour diagnostiquer les blocages inattendus. Cette rigueur est directement transposable à tout équipement réseau professionnel.

Analyse Réflexive

Le Cycle de Gibbs est un modèle de réflexion structurée qui me permet d'analyser en profondeur mes expériences d'apprentissage et d'améliorer continuellement ma pratique.

1
Étape 1

Description des faits

Décrire objectivement ce qui s'est passé, sans jugement.

Lors de la SAÉ Cyber.03 (Semestre 3), j'ai dû concevoir seul et déployer une infrastructure réseau complète pour moderniser le système d'information d'une entreprise fictive. En 4 semaines, il m'a fallu virtualiser un Windows Server 2019 sur vCenter, configurer l'Active Directory (domaine etu7.sae), mettre en place OSPFv2 sur du matériel Cisco physique, déployer un réseau Wi-Fi WPA2-Enterprise avec authentification RADIUS et publier des applications RemoteApp. En parallèle, j'ai réalisé des travaux de veille technologique (choix Grafana vs Zabbix, Ubiquiti vs Cisco).

2
Étape 2

Investigation du ressenti

Exprimer honnêtement vos émotions et ressentis subjectifs.

Au démarrage, j'ai ressenti un mélange d'enthousiasme (projet ambitieux et technique) et d'inquiétude face à l'ampleur du périmètre à couvrir seul. La multiplication des technologies à maîtriser simultanément (vCenter, AD, OSPF, RADIUS, RDS) a généré du stress, notamment lors du blocage causé par le conflit de port 8080 sur le contrôleur Unifi. La résolution méthodique de cet incident m'a procuré une vraie satisfaction et un regain de confiance. En fin de projet, obtenir la connexion Wi-Fi WPA2-Enterprise validée avec le compte Active Directory a été un moment de véritable fierté.

3
Étape 3

Analyse et évaluation

Évaluer objectivement ce qui a bien fonctionné et ce qui aurait pu être amélioré.

Points positifs : Ma démarche de diagnostic face au conflit de port Unifi (analyse des logs → identification du PID → arrêt ciblé) a été efficace et m'a évité une perte de temps. Les choix technologiques justifiés par des matrices de décision ont démontré une capacité à raisonner objectivement plutôt que par habitude. Le cloisonnement réseau (ACL, VLAN, réseaux visités isolés) était fonctionnel.

Points à améliorer : j'aurais dû anticiper davantage les dépendances entre services (ex. : le DNS doit être opérationnel avant de promouvoir le Contrôleur de Domaine). Une meilleure planification initiale via un diagramme de dépendances aurait évité plusieurs allers-retours.

4
Étape 4

Conclusions

Tirer des enseignements généraux et personnels de cette expérience.

Ce projet m'a appris que l'autonomie technique s'acquiert en acceptant de se confronter à l'inconnu sans chercher à tout maîtriser a priori. J'ai découvert que ma capacité à diagnostiquer méthodiquement un incident (plutôt que de réinstaller "à l'aveugle") est un point fort réel. En revanche, je dois progresser sur la planification structurée des déploiements complexes : documenter les dépendances avant de commencer évite des cycles d'erreur coûteux en temps.

5
Étape 5

Retour à l'action

Définir des actions concrètes pour progresser.

Pour mes prochains projets techniques, je vais systématiquement rédiger un plan de dépendances avant tout déploiement. Je vais approfondir la sécurité réseau Fortinet (SAÉ4.Cyber.01 en cours) et me préparer à la certification Cisco CyberOps Associate (déjà entamée à l'étape 2 de Cyber.04). Sur le plan du travail collaboratif, j'appliquerai le modèle de matrice de décision (Grafana vs Zabbix, Ubiquiti vs Cisco) à d'autres choix technologiques pour continuer à développer une argumentation technique rigoureuse et documentée.

Réflexion globale – S3 & S4

1

Ce que j'ai fait

Cinq SAÉ sur deux semestres, dans des domaines qui n'ont pas grand-chose en commun : IoT LoRaWAN sur VPS, application Android avec RBAC, infrastructure AD/RADIUS/vCenter, tests d'intrusion sur VulnHub, et sécurité Fortinet avec serveur Bastion. À chaque fois, j'ai dû apprendre un environnement que je ne connaissais pas encore — souvent en partant d'une page blanche.

2

Ce que j'ai ressenti

La progression n'a pas été linéaire. Les SAÉ techniques (Cyber.03, Pentesting) m'ont mis en difficulté, mais pas de la même façon : l'infra réseau demandait de la rigueur sur les dépendances, le pentesting demandait d'accepter de chercher sans trouver tout de suite. Les deux m'ont appris que l'inconfort est souvent le signe qu'on apprend vraiment.

3

Ce que j'analyse

Mon profil se clarifie : je suis à l'aise dans les rôles qui mêlent technique et méthode — diagnostiquer un incident, argumenter un choix technologique, documenter proprement. Ce qui reste à travailler, c'est la planification en amont des projets complexes et la gestion du temps quand plusieurs technologies se croisent.

4

Ce que j'en conclus

Ces deux semestres ont confirmé que la cybersécurité est le domaine où je veux aller, défense comme offense. Mais la technique seule ne suffit pas — savoir documenter, expliquer et justifier ses choix compte autant en entreprise que savoir configurer un FortiGate.

5

Mon plan d'action

Le stage qui commence le 6 avril 2026 est la prochaine étape concrète. L'objectif n'est pas juste de valider les crédits : c'est de trouver un contexte où ce que j'ai appris devient opérationnel sur des problèmes réels. Je cherche une entreprise où je peux travailler sur de la sécurité réseau avec des gens qui ont des choses à m'apprendre — et où je peux enchaîner en alternance dès septembre. La certification Cisco CyberOps Associate avance en parallèle pour structurer les connaissances offensives.
